一、项目总览

「匣中珠光」是一款以中国传统螺钿工艺为主题的浏览器互动剧情游戏。玩家扮演一位年轻学徒，在老匠人的指导下体验五道螺钿工序，与阿莲交流，最终完成一件螺钿作品。游戏采用纯前端单文件架构，无任何框架依赖，包含完整的对话分支、五个 Canvas 小游戏、成就系统、知识卡片、造物模式和 NG+ 二周目。

二、技术架构
🏗️架构概览

整个游戏采用单 HTML 文件 + IIFE 封装架构。CSS（约 287 行）、HTML DOM（约 68 行）、JavaScript（约 1906 行）全部内联。全局状态对象 G 挂在 window 上，通过显式导出暴露必要接口供调试和测试使用。

🎨渲染层

Canvas 2D API 承担所有场景渲染。核心优化手段是离屏静态缓存（Offscreen Canvas）：将复杂的木墙、窗户、工作台等静态元素渲染到独立 Canvas，主循环仅执行 drawImage 一次 blit。动态层（角色、粒子、光线）叠加在静态底图上。DPR 自适应（上限 2x）确保高分屏清晰度，防抖 resize（150ms）避免频繁重绘。

🔊音频层

完全基于 Web Audio API 的程序化音频系统，无需任何音频文件。AudioSys 生成 4 种音效（click/success/fail/swoosh），MusicSys 实现程序化五声音阶背景音乐，含 4 种 mood（title/prologue/craft/ending）、暖色 waveshaper、并联延迟混响和自适应 tempo。信号链为：oscillator → gain → master → lowpass(2400Hz) → waveshaper → destination + delay feedback(340ms/26%)。

💾持久化层

基于 localStorage 的三级存储：自动存档（xzzg_save_v3）、3 个手动存档槽（xzzg_save_v3_1/2/3）、设置（xzzg_settings）和成就（xzzg_achievements_v1）。存档内容涵盖所有流程状态、工艺质量评分、已读对话和收集进度。

✨粒子引擎

对象池模式的 2D 粒子系统，支持 7 种粒子类型（尘埃、萤火虫、光晕、贝壳火花、闪烁、花瓣），池上限 50，活跃数根据屏幕面积和画质设置动态调整（12-30）。含风力模拟（正弦波风场）、萤火虫双层闪烁、贝壳火花 HSL 色相偏移等特殊效果。

♿无障碍支持

全面使用 ARIA 属性：role（region/dialog/menu/toolbar/group/img）、aria-label、aria-live="polite"、aria-hidden 覆盖所有交互元素。Canvas 元素带有描述性 aria-label。在同类单文件游戏中极为少见。

五、引擎选型分析
结论：不建议引入游戏引擎
当前项目已高度成熟，引入引擎的迁移成本远大于收益。以下是各引擎的详细评估。
Phaser 3
功能过剩。Phaser 擅长精灵动画、物理引擎、tilemap，而本项目以静态场景+对话为主，小游戏逻辑高度定制化。迁移需重写全部 Canvas 渲染和小游戏交互，约 1900 行 JS 需要重构。
Cocos Creator
学习曲线陡峭。编辑器+TypeScript 工作流与当前纯 HTML/JS 项目差异巨大。打包体积大，对单文件部署的 Web 游戏来说是负担。
Godot
方向不符。Godot 是通用 2D/3D 引擎，GDScript 语言需要完全重写。Web 导出体积大（~15MB WASM），失去零依赖、即开即玩的核心优势。
PixiJS
可以考虑但非必要。PixiJS 是轻量渲染库（非完整引擎），WebGL 加速可提升粒子性能。但当前 Canvas 2D 性能已足够，引入后仍需自建对话/音频/存档框架。
保持现状 ✓
零依赖、即开即玩、单文件部署是本项目核心竞争力。所有功能已完善，架构稳定。建议继续在现有架构上迭代。
⚖️迁移成本对比
维度当前架构迁移至引擎
代码重写量0~1900 行（几乎全部 JS）
学习成本无1-3 个月
部署体积<100KB500KB-15MB

六、音乐系统评估
🎵当前音乐系统

MusicSys 是一个精巧的程序化音乐生成器，基于 Web Audio API 实现。它使用五声音阶（宫/商调式）、暖色 waveshaper 模拟木质共鸣、并联延迟反馈（340ms/26%）和 4 种 mood 配置。每个音随机选择音阶度数、八度和时长，40% 概率叠加低八度三角波衬底，25% 概率叠加高八度音。tempo 自适应（基础间隔 × 0.6~1.4 随机因子）。

🎼音乐升级方案
方案优势劣势推荐度
保持程序化生成零文件体积、无限变化、技术亮点音色单一、缺乏情感层次★★★★
加载 MIDI + Tone.js可用真实音色、文件小（~50KB）需引入 Tone.js 库（~400KB）★★★
加载 MP3/OGG 音频最高音质、情感表达力强文件体积大（1-5MB）、破坏零依赖★★
增强当前系统保持零依赖、提升表现力需要音频 DSP 知识★★★★★

★ 推荐方案：增强当前程序化系统

可在不引入外部文件的前提下提升音乐表现力：增加和弦进行（I-IV-V-I）、添加琶音伴奏层、引入简单旋律片段（4-8 小节循环）、使用 ADSR 包络替代固定 gain 控制、添加 tremolo/vibrato 调制。这些改进可将程序化音乐从「氛围音」提升至「有旋律感的背景音乐」级别。

七、优化建议
1. 程序化音乐增强
高优先
当前 MusicSys 生成的是随机音阶氛围音，缺乏旋律记忆点。建议增加：ADSR 包络控制（attack/decay/sustain/release）使音符更自然；基础和弦进行（如 C-Am-F-G 五声变体）为每个 mood 建立和声骨架；4-8 小节旋律片段（预定义音程序列）在随机音之间穿插，形成主题感；tremolo 调制（LFO 控制 gain）为长音添加颤动效果。
效果：背景音乐从「环境音」提升至「有辨识度的主题旋律」，玩家可通过音乐感知场景情绪变化，增强沉浸感。
2. 角色表现力增强
高优先
当前角色为静态简笔画 + 表情切换。建议增加：说话时身体微幅摇摆（正弦波驱动 torso 偏移）；手势动作（挥手/抱拳/指引，关键对话时触发）；眨眼动画（随机间隔 3-7 秒，闭眼 200ms）；对话情绪色彩光环（悲伤时角色周围淡蓝色光晕，喜悦时暖黄色光晕）。这些都可以用纯 Canvas 实现，无需外部资源。
效果：角色从「符号化图标」变成「有生命力的角色」，对话场景的情感传达大幅提升。对 TRAE 评审的视觉印象深刻。
3. 小游戏难度递进可视化
中优先
当前五个小游戏的评价阈值是固定的。建议添加：动态难度提示（首次玩显示「轻松模式」/「挑战模式」选择）；连续优秀时解锁「匠人挑战」（更严格阈值但额外成就奖励）；小游戏回放功能（完成后可观看自己的操作轨迹）；每个小游戏的最佳成绩记录与排行榜（localStorage 存储 top 3）。
效果：增加小游戏的重玩价值和挑战深度，为高水平玩家提供额外目标。
4. 场景氛围时间系统
中优先
当前场景光照是静态的。建议引入基于章节的「时间流逝」系统：序章为清晨（淡金色光线从左窗投入），中段为午后（光线从上方直射），告别场景为黄昏（橙红色渐变+长阴影），结局为月夜（银蓝色调+月光束）。通过在不同章节切换 buildPrologueStatic 中的窗户光线参数和叠加色彩滤镜即可实现。
效果：玩家通过光线变化感知时间推移，增强叙事沉浸感。「月夜」结局与螺钿「月光映照」主题完美呼应。
5. 对话系统增强
中优先
建议添加：对话回放功能（结局后可在图鉴中重新体验已解锁的关键对话）；角色立绘差分（同一角色不同情绪的 Canvas 绘制预设，关键剧情时切换）；对话标记系统（玩家可标记喜欢的对话，存储在 localStorage，图鉴中设「收藏」标签页展示）；打字机效果增加标点停顿（逗号 100ms、句号 200ms、问号/叹号 300ms），使阅读节奏更自然。
效果：对话体验从「阅读文本」提升为「体验故事」，标点停顿让叙事节奏更舒适。
6. 代码架构优化
中优先
当前 2261 行单文件虽然功能完整，但维护成本较高。建议（在不改变单文件部署的前提下）：使用注释分隔逻辑模块（如 // ═══ Module: Dialogue ═══）；将各小游戏封装为独立对象（MiniGames.tracing、MiniGames.cutting）；状态管理引入简单的事件系统（状态变更时触发回调，而非散落的 if 判断）；CSS 使用 CSS 自定义属性（--color-gold: #d4a853）替代硬编码色值。
效果：代码可读性和可维护性大幅提升，后续添加新功能时不会破坏已有逻辑。对 TRAE 代码评审有直接加分。
7. 造物模式扩展
中优先
当前造物模式支持 3×3×3 = 27 种组合。建议扩展：增加自定义纹样绘制（简化版拓样小游戏，在造物模式中自由绘制纹样）；作品保存为 PNG 下载（canvas.toDataURL()）；作品画廊（localStorage 存储历史作品缩略图，最多 12 件）；增加更多漆色选项（青瓷绿、象牙白、墨玉黑）和器物形状（盘、碗、屏风）。
效果：造物模式从「预览工具」变成「创作沙盒」，大幅提升游戏的创意表达价值和社交分享潜力。
8. 环境音效层
低优先
在现有 AudioSys 基础上添加环境音效层：工坊场景有轻微的木工声（低频噪声脉冲）、窗外的风声（滤波白噪声）、远处的鸟鸣（高频正弦波随机触发）。这些效果可完全通过 Web Audio API 生成，无需音频文件。每种环境音效可设置独立的音量和开关。
效果：营造更立体的空间感，工坊场景不再是「静音画面+背景音乐」，而是有声音质感的真实空间。
9. 移动端体验优化
低优先
当前已有触摸支持和响应式 CSS，但可进一步优化：小游戏 Canvas 在移动端的自适应尺寸（当前固定 460×260）；添加触觉反馈（navigator.vibrate()，在成就解锁、小游戏评价等关键时刻触发）；PWA 支持（Service Worker + manifest，实现离线可玩和添加到主屏幕）；横屏/竖屏自适应布局切换。
效果：移动端玩家获得与桌面端同等流畅的体验，PWA 支持让游戏可像原生 App 一样使用。
10. 统计与数据可视化
低优先
在图鉴或成就面板中增加游戏统计：总游玩时间、每个小游戏的平均得分分布（柱状图，Canvas 绘制）、对话选择偏好（饼图）、结局达成进度（环形图，4 种结局完成百分比）、记忆碎片收集时间线。全部使用 Canvas 2D 绘制，无需引入 Chart.js 等库。
效果：为玩家提供「游戏回顾」的满足感，也是 TRAE 评审中数据可视化能力的展示。
11. 国际化支持（i18n）
中优先
